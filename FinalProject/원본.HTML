  • 데이터 정규화 (Normalization):상점(ShopPage)과 인벤토리(InventoryPage)에서 서버 데이터 필드명이 다르더라도(itemCategory vs category) 렌더링 직전에 정규화하여 공통 컴포넌트가 일관된 인터페이스로 동작하게 했습니다.

Code Point

상점, 인벤토리, 모달창 등 어디서든 이 코드 한 줄로 아이템의 비주얼을 결정합니다.

Auth & Lifecycle

  • 인증 라이프사이클 설계: 회원가입부터 로그인, 로그아웃에 이르는 사용자 인증의 전 과정을 RESTful API로 설계하고 핵심 비즈니스 로직을 구현함.

  • Soft Withdrawal(논리 삭제) 시스템: 회원 탈퇴 시 데이터를 즉시 삭제하지 않고 상태값 변경을 통해 관리함으로써 데이터 무결성 및 사후 복구 가능성 고려.

코드 리뷰: 회원 가입 및 JWT 로그인 로직

// 1. 회원 가입 시 비밀번호 암호화 및 필수값 검증 @PostMapping("/join") public ResponseEntity<?> insertMember(@RequestBody MemberVO m){ if(m.getLoginId() == null || m.getPassword() == null) { return ResponseEntity.badRequest().body("아이디와 비밀번호는 필수입니다."); } m.setPassword(bcrypt.encode(m.getPassword())); // 암호화 처리 int result = service.insertMember(m); return result > 0 ? ResponseEntity.status(HttpStatus.CREATED).body("가입 성공") : ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body("가입 실패"); } // 2. JWT 기반 보안 로그인 로직 @PostMapping("/login") public ResponseEntity<?> loginMember(@RequestBody MemberVO m) { MemberVO loginMember = service.loginMember(m); if (loginMember == null || !bcrypt.matches(m.getPassword(), loginMember.getPassword())) { return ResponseEntity.status(HttpStatus.UNAUTHORIZED).body("인증 실패"); } String token = jwtUtil.generateToken(loginMember.getLoginId()); // 토큰 생성 loginMember.setPassword(null); // 보안을 위해 비번 필드 제거 HashMap<String, Object> map = new HashMap<>(); map.put("token", token); map.put("user", loginMember); return ResponseEntity.ok(map); }

// ChatRoomDetail.jsx - 구독 수신부 로직 const roomSubscription = client.subscribe(`/topic/chat/room/${roomId}`, (message) => { const msg = JSON.parse(message.body); // [핵심] 단일 통신망 안에서 메시지 Type에 따른 프론트엔드 역할 분기 처리 switch (msg.type) { // 1. 일반 채팅 (Data Layer) -> 로컬 배열 상태 갱신 case 'TEXT': case 'IMAGE': setMessages(prev => [...prev, msg]); // 상대가 보낸 메시지라면 즉각 읽음 처리 동기화 if (msg.senderId !== user.memberId) markAsRead(msg.messageId); break; // 2. 시스템 안내 멘트 (Notice Layer) -> 화면 상단 공지 렌더링 case 'SYSTEM': setSystemNotice(msg.content); break; // 3. 메시지 삭제 로컬 동기화 (Sync Layer) -> 컴포넌트 실시간 제거 case 'DELETED': setMessages(prev => prev.map(m => m.messageId === msg.messageId ? { ...m, isDeleted: true } : m)); break; // 4. 무음 상태 제어 신호 (Control Layer) -> 백그라운드 참여자 목록 갱신 case 'ROOM_UPDATE': fetchRoomMembers(); break; } });

  • 단일 터널 다형성 제어:단 하나의 채널만을 구독하고, 백엔드 DTO의 type 필드를 식별자로 삼아 일반 채팅, 안내 멘트, 방 제어 등 완전히 다른 성격의 UI/UX 업데이트를 스위칭합니다.
  • 무음 제어 신호 설계:ROOM_UPDATE 타입은 화면에 빈 말풍선을 띄우지 않는 **투명 제어 신호**입니다. 해당 타입 수신 시 백그라운드 API만 찔러 우측 채팅 참여자 목록을 조용히 갱신합니다.

// ChatRoomDetail.jsx const handleSend = () => { const msgDto = { chatRoomId: roomId, senderId, content: input, messageType: 'TEXT' }; // [중요] 낙관적 업데이트 (Optimistic Update) const tempId = Date.now(); const optimisticMsg = { ...msgDto, messageId: tempId, isOptimistic: true }; // 1. 서버 전송 전, 내 화면에 먼저 그리기 (체감속도 단축) setMessages(prev => [...prev, optimisticMsg]); try { // 2. 백엔드 컨트롤러(/app/chat/message)를 향해 발행 (Pub) client.publish({ destination: '/app/chat/message', body: JSON.stringify(msgDto) }); setInput(''); } catch (error) { // 전송 실패 시 임시 그려둔 말풍선 제거 롤백 setMessages(prev => prev.filter(msg => msg.messageId !== tempId)); } };

Code Point

  • 메시지 발행 (client.publish):규격화된 DTO 포맷에 맞추어 송신 데이터를 백엔드 라우팅 경로(/app)로 발사합니다.
  • 낙관적 업데이트 (Optimistic Update)와 동기화:서버 응답을 기다리지 않고 임시 ID를 가진 말풍선을 화면에 먼저 렌더링하여 체감 지연(Delay)을 없앴습니다. 이때 임시 객체에 isOptimistic: true 플래그를 달아두고, 직후 서버로부터 진짜 messageId가 포함된 웹소켓 응답이 돌아왔을 때 플래그를 추적하여 새 말풍선을 중복 추가하는 대신 진짜 데이터로 덮어씌우는(Replace) 맵핑 동기화 로직을 구현해 무결성을 보장했습니다.

  • 실시간 브로드캐스트:프론트에서 전송받은 메시지를 ChatController가 받아 Service 계층을 통해 DB에 안전하게 저장한 직후, 해당 방 전체에 messagingTemplate을 통해 0.1초 만에 최신화된 메시지를 방송합니다.
  • 트랜잭션 기반 동기화 (무결성 보장):채팅방 외부에 있는 유저들의 전역 알림(Unread Count) 상태 역시 실시간으로 동기화해야 합니다. 컨트롤러 요청으로 서비스 계층에서 이벤트를 발행(Publish)하면, @TransactionalEventListener가 DB 커밋이 100% 확정된 순간만을 감지하여 알림 이벤트를 발송합니다. 이를 통해 알림 발송과 실제 데이터 저장 시점의 불일치로 인한 유령 메시지 버그를 원천 차단했습니다.

Code Point

  • @Cacheable (조회 성능 극대화):수많은 유저가 반복적으로 껐다 켰다 하는 '채팅방 상세 정보' 조회 쿼리를 DB까지 보내지 않습니다. 스프링이 관리하는 메모리 chatRoomDetails 저장소에서 바로 객체를 반환하므로 부하가 0에 가깝게 줄어듭니다.
  • @CacheEvict (캐시 동기화 전략):성능은 빨라졌지만 과거 데이터가 보이는 버그를 막아야 합니다. 채팅방 정보가 변하는 주요 액션(메시지 전송, 사용자 입퇴장 등) 길목마다 @CacheEvict 폭파 스위치를 심어두어, 즉시 기존 캐시를 증발시키고 다음번에 새로 가장 신선한 쿼리를 날려 캐시를 갱신하도록 설계했습니다.

// ReportsServiceImpl.java @Transactional public int reportsStatus(int reportsId, String status) { int result = dao.reportsStatus(sqlSession, reportsId, status); // RESOLVED일 때만 블라인드 체크 트리거 if(result > 0 && "RESOLVED".equals(status)) { ReportsVO report = dao.reportsDetail(sqlSession, reportsId); Map<String, Object> map = new HashMap<>(); map.put("type", report.getType()); // POST / REPLY / REVIEW map.put("postId", report.getPostId()); map.put("replyId", report.getReplyId()); map.put("reviewId", report.getReviewId()); reportsBlind(map); } return result; } public int reportsBlind(Map<String, Object> map) { // 누적 처리 횟수 조회 int resolvedCount = dao.selectResolvedCount(sqlSession, map); if (resolvedCount >= 10) { // type 하나로 POST·REPLY·REVIEW 세 테이블 동적 분기 return dao.reportsBlind(sqlSession, map); } return 0; }

Code Point

  • 자동화된 블라인드:관리자가 RESOLVED 처리하는 순간 누적 횟수를 자동 집계, 기준 이상이면 별도 조작 없이 블라인드 처리
  • 단일 분기 설계:게시글·댓글·리뷰 세 개의 다른 테이블을 type 값 하나로 map에 담아서 동적 분기, 중복 코드 없이 처리
  • 트랜잭션 보장:@Transactional로 상태 변경과 블라인드 처리를 하나의 단위로 묶어 데이터 정합성 유지
